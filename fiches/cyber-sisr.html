1
PowerShell

PowerShell est un shell en ligne de commande et un langage de script développé par Microsoft. Basé sur le framework .NET, il permet d'automatiser l'administration des systèmes Windows (et Linux depuis PowerShell Core).

Caractéristiques clés

  • Cmdlets : commandes natives au format Verbe-Nom (ex : Get-Service, Set-Item)
  • Pipeline (|) : enchaîner des commandes pour passer les objets d'une commande à l'autre
  • Scripts .ps1 : fichiers de scripts PowerShell exécutables
  • Objets .NET : contrairement au bash, PowerShell manipule des objets structurés, pas juste du texte

Commandes essentielles

CommandeDescription
Get-Help <cmd>Affiche l'aide d'une commande
Get-ServiceListe les services Windows
Start/Stop-ServiceDémarrer/arrêter un service
Get-ProcessListe les processus en cours
Get-EventLogConsulter les journaux d'événements
Get-ADUserGestion Active Directory (module RSAT)
Invoke-CommandExécuter des commandes sur un PC distant
Set-ExecutionPolicyDéfinir la politique d'exécution des scripts

ExecutionPolicy :
• Restricted : aucun script autorisé (par défaut)
• RemoteSigned : scripts locaux OK, distants doivent être signés
• Unrestricted : tous les scripts s'exécutent (risque sécurité)

Avantages

  • Automatisation puissante (création comptes AD, déploiement...)
  • Intégration native avec l'écosystème Microsoft (AD, Exchange, Azure)
  • Manipulation d'objets .NET : résultats structurés et filtrables
  • Multiplateforme depuis PowerShell Core
  • Télégestion : Invoke-Command sur des dizaines de serveurs
  • Traçabilité : scripts auditables et reproductibles

Inconvénients

  • Courbe d'apprentissage (syntaxe objets et pipelines)
  • Politique d'exécution restrictive par défaut
  • Risque de sécurité si mal sécurisé (scripts malveillants)
  • Moins adapté que bash pour les environnements Linux purs

Questions types d'examen

Q1. Automatiser la création de 50 comptes AD depuis un fichier CSV ?

→ Import-Csv + New-ADUser en boucle foreach. Module ActiveDirectory (RSAT).

Q2. Erreur « L'exécution de scripts est désactivée sur ce système » ?

→ ExecutionPolicy Restricted par défaut → Set-ExecutionPolicy RemoteSigned.

Q3. Vérifier à distance l'état des services sur 10 serveurs ?

→ Invoke-Command -ComputerName + Get-Service. Prérequis : WinRM activé, pare-feu, droits admin.

Q4. Différence entre PowerShell et cmd.exe ?

→ PowerShell = objets .NET, cmdlets, scripting avancé. CMD = commandes simples, compatibilité ancienne.

Q5. Lister les comptes admin locaux de 20 postes ?

→ Invoke-Command + Get-LocalGroupMember 'Administrators'. Export vers CSV avec Export-Csv.

2
RAID (Redundant Array of Independent Disks)

Le RAID est une technologie permettant de combiner plusieurs disques durs physiques en un seul volume logique pour améliorer les performances, la redondance (tolérance aux pannes) ou les deux.

Les niveaux de RAID principaux

NiveauDescription
RAID 0 — StripingDonnées réparties sur 2+ disques. Performances max. Aucune redondance : si 1 disque tombe, tout est perdu.
RAID 1 — MirroringDonnées dupliquées sur 2 disques identiques. Tolérance : 1 disque. Capacité utile = 1 seul disque.
RAID 5 — Striping + paritéDonnées + parité réparties sur 3+ disques. Tolérance : 1 disque. Capacité = (N-1) disques.
RAID 6 — Double paritéComme RAID 5 mais 2 disques de parité. Tolérance : 2 disques. Nécessite 4+ disques.
RAID 10 (1+0)Miroir de stripes. Hautes performances + tolérance aux pannes. Nécessite 4+ disques.

RAID matériel : contrôleur dédié, plus performant, transparent pour l'OS.
RAID logiciel : géré par l'OS (Windows Storage Spaces, mdadm Linux), moins performant.

Avantages

  • RAID 1 : tolérance immédiate, lecture rapide en parallèle
  • RAID 5 : bon compromis capacité/performance/redondance
  • RAID 10 : meilleures perfs lecture/écriture + haute tolérance
  • Continuité de service : le système continue sans interruption
  • Hot spare : disque de rechange à chaud automatique

Inconvénients

  • RAID 0 : aucune tolérance, perte totale si 1 disque tombe
  • RAID 5 : vulnérable lors de la reconstruction (2e panne = perte totale)
  • RAID n'est PAS une sauvegarde : suppression ou ransomware affecte tous les disques
  • RAID 5/6 : pénalité en écriture (calcul de parité)
  • RAID 10 : nécessite 2x plus de disques

Questions types d'examen

Q1. 4 disques de 2 To, maximiser disponibilité et performances ?

→ RAID 10 → 4 To utiles (50% de 8 To). 1 panne par miroir tolérée.

Q2. Un disque tombe en RAID 5 (3 disques de 1 To), conséquences ?

→ Fonctionnement dégradé. Si 2e panne avant reconstruction → perte totale.

Q3. « J'ai du RAID 1, donc mes données sont sauvegardées » ?

→ RAID ≠ sauvegarde. Ne protège pas contre suppression, ransomware, vol. Recommander 3-2-1.

Q4. RAID 5 vs RAID 6 ?

→ RAID 6 = 2 disques de parité, tolère 2 pannes. Choisir quand disques >4 To (reconstruction longue).

Q5. RAID matériel vs RAID logiciel ?

→ RAID matériel = contrôleur dédié, cache, meilleure perf, transparent. Recommandé en production.

3
HTTPS (HyperText Transfer Protocol Secure)

HTTPS est la version sécurisée du protocole HTTP. Il utilise TLS (Transport Layer Security) pour chiffrer les communications entre le client et le serveur web. Port par défaut : 443.

Fonctionnement TLS

  • 1. Handshake TLS : négociation des algorithmes + authentification via certificat
  • 2. Échange de clés : clé de session symétrique générée
  • 3. Communication chiffrée : toutes les données sont chiffrées

Types de certificats SSL/TLS

TypeDescription
DVDomain Validation — vérifie uniquement le domaine. Rapide et gratuit (Let's Encrypt).
OVOrganization Validation — vérifie l'organisation. Plus de confiance.
EVExtended Validation — nom de l'entreprise visible. Très haute confiance.
Auto-signéNon reconnu publiquement. Usage interne uniquement.

HTTP : données en clair, port 80, vulnérable. HTTPS : données chiffrées, port 443, authentification du serveur. Un site phishing peut aussi avoir un certificat HTTPS valide !

Avantages

  • Chiffrement de bout en bout
  • Authentification du serveur via certificat
  • Intégrité des données garantie par TLS
  • SEO : Google favorise les sites HTTPS
  • Gratuit avec Let's Encrypt (certificats DV)

Inconvénients

  • Certificats OV/EV payants, renouvellement à gérer
  • Un certificat valide ne garantit PAS la légitimité du site
  • Config incorrecte (TLS 1.0/1.1, cipher faibles) = moins sécurisé

Questions types d'examen

Q1. Mettre en place HTTPS sur un serveur Apache/Nginx ?

→ Obtenir certificat (Let's Encrypt ou CA interne), configurer virtual host SSL, port 443, rediriger HTTP→HTTPS, HSTS.

Q2. URL 'www.cr-agricoIe.fr' (I majuscule au lieu de l) ?

→ Attaque homographe / typosquatting. HTTPS ne garantit pas la légitimité. Vérifier l'URL, utiliser les favoris.

Q3. Mécanisme de handshake TLS ?

→ Version TLS, suite cryptographique, échange de clés (DH/ECDH), authentification par certificat, clé de session.

Q4. Certificat auto-signé vs CA reconnue ?

→ Auto-signé : pas de tiers de confiance, avertissement navigateur, usage interne. CA reconnue : confiance publique.

Q5. Site web sous TLS 1.0, risques ?

→ TLS 1.0/1.1 vulnérables (POODLE, BEAST). Migrer vers TLS 1.2 minimum, idéalement 1.3.

4
SSH (Secure Shell)

SSH est un protocole réseau cryptographique permettant de se connecter à distance à un serveur de façon sécurisée. Il remplace Telnet (non chiffré). Port par défaut : 22.

Commandes essentielles

ssh user@ip                    # Connexion SSH basique
ssh -p 2222 user@ip            # Port non standard
ssh -i clé.pem user@ip         # Connexion avec fichier de clé privée
scp fichier user@ip:/chemin    # Copier un fichier vers un serveur distant
sftp user@ip                   # Session de transfert interactive

Authentification par clé SSH :
1. ssh-keygen (générer la paire de clés)
2. ssh-copy-id user@serveur (copier la clé publique)
3. La clé privée reste sur le client, ne quitte jamais la machine

Bonnes pratiques SSH : Changer le port (ex : 2222) | PermitRootLogin no | Désactiver l'auth par mot de passe (PasswordAuthentication no) | Fail2Ban pour bloquer les IP après trop de tentatives

Avantages

  • Toutes les communications sont chiffrées
  • Authentification forte par clé
  • Multiusage : accès distant, SCP/SFTP, tunneling
  • Léger et disponible partout
  • Traçabilité : connexions loguées

Inconvénients

  • Port 22 très ciblé (scanners, bots bruteforce)
  • Gestion des clés complexe à grande échelle
  • Si clé privée compromise sans passphrase → accès immédiat
  • Pas d'interface graphique natif (CLI uniquement)

Questions types d'examen

Q1. Sécuriser l'accès SSH d'un serveur Linux (4 mesures) ?

→ Port non standard, PermitRootLogin no, PasswordAuthentication no (clés uniquement), AllowUsers, MaxAuthTries.

Q2. Processus d'authentification par clé publique/privée ?

→ ssh-keygen + ssh-copy-id. Le serveur chiffre un défi avec la clé pub, le client déchiffre avec la clé privée.

Q3. Centaines de tentatives SSH par jour ?

→ Analyser /var/log/auth.log. Fail2Ban, changer port, désactiver auth par mot de passe, pare-feu.

Q4. SCP vs SFTP ?

→ SCP = copie simple, rapide, scriptable. SFTP = session interactive, navigation, reprise sur erreur.

Q5. SSH vers les équipements Cisco depuis le siège ?

→ Bastion SSH (jump host), clés SSH, IP whitelisting, config AAA sur Cisco (ip ssh version 2).

IDS et IPS

IDS (Intrusion Detection System) : surveille le trafic et génère des alertes. Il ne bloque pas, il observe.
IPS (Intrusion Prevention System) : détecte ET bloque activement le trafic malveillant en temps réel. Positionné en inline dans le flux.

CritèreIDSIPS
RôleDétecte uniquementDétecte ET bloque
PositionMode passif (copie du trafic)Mode inline (flux réel)
RéactionGénère des alertesBloque, rejette, réinitialise
Faux positifsFaible impactPeut bloquer du trafic légitime
ExemplesSnort (mode IDS), SuricataSnort (mode IPS), Suricata IPS

Types de détection

  • Par signature : compare à une base de signatures connues (efficace pour menaces connues)
  • Comportementale / anomalie : profil normal puis alerte en cas de déviation (efficace pour zero-day)
  • NIDS/NIPS : surveille le trafic réseau global
  • HIDS/HIPS : surveille les activités sur un seul système (fichiers, processus, logs)

Outils : Snort, Suricata (NIDS/NIPS) | OSSEC, Wazuh (HIDS) | Fail2Ban (bannissement d'IP simple)

Avantages

  • Détection en temps réel des attaques
  • Visibilité réseau complète + analyses forensiques
  • IPS : blocage automatique sans intervention humaine
  • Détection comportementale : détecte les menaces zero-day
  • Complémentaire au pare-feu (analyse le contenu)

Inconvénients

  • Faux positifs : risque de bloquer du trafic légitime (IPS)
  • Faux négatifs : attaque sophistiquée ou chiffrée peut passer
  • Signatures à mettre à jour régulièrement
  • DPI peut introduire de la latence
  • HTTPS chiffré difficile à inspecter

Questions types d'examen

Q1. Différence IDS / IPS, lequel recommander en premier ?

→ IDS d'abord pour phase d'apprentissage, puis IPS une fois les règles affinées (éviter les faux positifs).

Q2. Alertes 'SYN Flood détecté' dans les logs IDS ?

→ Attaque DDoS SYN Flood. Contre-mesures : SYN cookies, limitation connexions, filtrage IPS, blackholing.

Q3. Différence NIDS et HIDS ?

→ NIDS = réseau (Snort). HIDS = hôte (OSSEC). Combiner : NIDS pour le périmètre, HIDS pour les serveurs critiques.

Q4. Détection par signature vs par anomalie ?

→ Signature : efficace menaces connues, nécessite MAJ. Anomalie : détecte zero-day, plus de faux positifs.

Q5. IPS bloque des connexions légitimes vers la messagerie ?

→ Analyser logs IPS, identifier la règle, vérifier si faux positif, créer exception (whitelist), documenter.

6
Attaque Homme du Milieu (Man-in-the-Middle)

Une attaque MitM consiste pour un attaquant à s'intercaler secrètement entre deux parties qui communiquent. Il peut écouter, modifier ou injecter des données à l'insu des victimes.

Techniques d'attaque

  • ARP Poisoning : fausses réponses ARP → tout le trafic passe par l'attaquant
  • DNS Spoofing : falsification des réponses DNS → redirection vers un faux site
  • SSL Stripping : dégrade HTTPS en HTTP → écoute des données en clair
  • Evil Twin (WiFi) : point d'accès WiFi malveillant imitant un réseau légitime
  • BGP Hijacking : détournement du routage Internet

Protections

  • HTTPS / TLS : chiffrer les communications
  • HSTS : force le navigateur à toujours utiliser HTTPS
  • VPN : chiffrer tout le trafic réseau
  • DAI (Dynamic ARP Inspection) : valide les paquets ARP sur les switchs managés
  • Éviter les WiFi publics ou utiliser un VPN systématiquement

Scénario type : un employé se connecte au WiFi d'un hôtel. Un attaquant a créé un faux point d'accès (Evil Twin). Toutes ses communications sont interceptées. Solution : VPN obligatoire sur tout réseau non maîtrisé.

Questions types d'examen

Q1. Suspicion d'ARP spoofing, comment détecter et corriger ?

→ Analyser table ARP (arp -a), arpwatch/XArp. Corrective : activer DAI sur switchs Cisco, DHCP snooping.

Q2. Commercial utilisant le WiFi d'un hôtel pour le CRM ?

→ Risques : Evil Twin, écoute réseau. Politique : VPN obligatoire, HTTPS pour toutes les apps, sensibilisation.

Q3. Attaque SSL Stripping, comment HSTS protège ?

→ SSL Stripping dégrade HTTPS→HTTP. HSTS : le serveur impose HTTPS au navigateur (Strict-Transport-Security header).

Q4. Email du PDG demandant un virement de 50 000€ ?

→ Business Email Compromise (BEC). Mesures : DMARC/DKIM/SPF, procédure double validation, MFA messagerie.

Q5. Qu'est-ce que le certificate pinning ?

→ Associer une app à un certificat spécifique → résiste aux faux certificats. Utilisé dans les apps mobiles.

ITIL (Information Technology Infrastructure Library)

ITIL est un référentiel de bonnes pratiques pour la gestion des services informatiques (ITSM). Cadre de référence (framework), pas une norme imposée. Version actuelle : ITIL 4.

Les 5 étapes du cycle de vie ITIL

ÉtapeDescription
1. Stratégie des servicesDéfinir la stratégie, aligner l'IT sur les objectifs métier
2. Conception des servicesConcevoir les nouveaux services (SLA, capacité, disponibilité, sécurité)
3. Transition des servicesMettre en production en maîtrisant les risques (gestion des changements, tests)
4. Exploitation des servicesGérer au quotidien (incidents, problèmes, demandes, accès)
5. Amélioration continueMesurer, analyser et améliorer en permanence (PDCA)

Incident vs Problème vs Changement :
• Incident : interruption non planifiée. Action immédiate. Ex : messagerie inaccessible.
• Problème : cause sous-jacente d'incidents. Investigation de fond. Ex : pourquoi le serveur tombe chaque lundi ?
• Changement : modification planifiée. Ex : mise à jour de l'OS du serveur.

Avantages

  • Alignement IT / métier
  • Standardisation des processus (même langage)
  • Amélioration de la qualité de service
  • Gestion des risques (gestion des changements)
  • SLA mesurables et suivis

Inconvénients

  • Lourdeur administrative pour les petites structures
  • Coût de mise en oeuvre (formation, outils ITSM)
  • Risque de bureaucratie si mal appliqué
  • Cadre à adapter, pas de recette unique

Questions types d'examen

Q1. 30 utilisateurs signalent que Outlook ne fonctionne pas. Processus ITIL ?

→ Détection, enregistrement, catégorisation, priorisation, diagnostic, escalade, résolution, clôture.

Q2. Lenteur réseau récurrente en fin de mois (comptabilité). Gestion des problèmes ?

→ Incident = rétablir. Problème = investiguer (RCA). Démarche : workaround, erreur connue (KEDB), solution définitive.

Q3. Migration serveur de fichiers vers un NAS. Gestion des changements ?

→ RFC, évaluation risques/impacts, validation CAB, plan de rollback, test pré-prod, déploiement, revue post-implémentation.

Q4. Qu'est-ce qu'un SLA ? Exemple concret ?

→ Accord de niveau de service. Ex : incidents critiques résolus en 4h, disponibilité 99,9%, hotline 8h-18h.

Q5. GLPI et démarche ITIL ?

→ GLPI = outil ITSM. Supporte : gestion incidents (tickets), gestion actifs/CMDB, contrats, base de connaissances.

8
Sauvegardes et Restaurations

Une sauvegarde (backup) est une copie des données permettant de les restaurer en cas de perte, corruption, cyberattaque ou sinistre. Élément central de la continuité d'activité.

Types de sauvegardes

TypeDescription
Complète (Full)Copie intégrale. Sûre et simple à restaurer, mais longue et volumineuse.
IncrémentaleUniquement les données modifiées depuis la DERNIÈRE sauvegarde. Rapide. Restauration complexe (full + toutes les incrémentales).
DifférentielleDonnées modifiées depuis la DERNIÈRE SAUVEGARDE COMPLÈTE. Restauration : full + dernière différentielle uniquement.
Miroir (Clone)Copie exacte synchronisée. Pas d'historique.

Règle 3-2-1 : 3 copies des données | 2 supports différents | 1 copie hors site

Concepts importants

  • RPO (Recovery Point Objective) : perte de données maximale acceptable. RPO = 4h → max 4h de perte
  • RTO (Recovery Time Objective) : temps maximal pour restaurer le service. RTO = 2h → service rétabli en <2h
  • PCA : Plan de Continuité d'Activité (maintenir pendant sinistre)
  • PRA : Plan de Reprise d'Activité (reprendre après sinistre)

Avantages

  • Protection contre tous types de pertes
  • Règle 3-2-1 : aucun événement unique ne détruit tout
  • Incrémentale : rapide quotidiennement
  • Test de restauration possible
  • RPO/RTO : objectifs mesurables

Inconvénients

  • Coût (stockage, licences, bande passante)
  • Restauration complète peut prendre des heures/jours
  • Incrémentale : restauration complexe (full + toutes dans l'ordre)
  • Sauvegardes non testées = non fiables
  • Sauvegardes connectées : vulnérables aux ransomwares

Questions types d'examen

Q1. Politique de sauvegarde 3-2-1 pour données patients ?

→ Full dimanche, incrémentales lun-sam. Copie NAS + bandes (coffre) + cloud chiffrée. Rétention 30 jours min.

Q2. RPO 4h et RTO 2h, signification concrète ?

→ RPO 4h = sauvegardes toutes les 4h. RTO 2h = infra redondante, procédures testées, serveur de secours.

Q3. Ransomware mardi 14h, dernière full dimanche, incrémentales lun et mar matin ?

→ Isoler le serveur, restaurer full (dimanche) + incrémentale lundi + incrémentale mardi matin. Perte = mardi matin à 14h.

Q4. Pourquoi tester les restaurations régulièrement ?

→ Sauvegarde non testée = infiable. Tests trimestriels min. PV de test : date, durée, volume, résultat, signature.

Q5. Migration sauvegardes vers le cloud ?

→ Avantages : hors site automatique, scalabilité. Risques : confidentialité (chiffrer), bande passante, RGPD localisation.

Force brute et attaque par dictionnaire
TypeMéthodeVitesse
Force bruteTeste toutes les combinaisons possiblesLent mais exhaustif
DictionnaireListe de mots de passe courants (rockyou.txt : 14M mots)Très rapide
HybrideDictionnaire + règles de transformation (password → P@ssw0rd)Efficace
Rainbow TableTables précalculées de hashesTrès rapide, contrée par le salage

Contre-mesures

  • Mots de passe forts : 12+ caractères, majuscules, minuscules, chiffres, spéciaux
  • Verrouillage de compte : bloquer après N tentatives échouées
  • MFA : second facteur même si le mdp est découvert
  • Fail2Ban / IPS : bloquer automatiquement les IP
  • Salage des mots de passe : valeur aléatoire avant hachage (contre les rainbow tables)

Questions types d'examen

Q1. Base de données de hashes MD5 extraite, risques et actions ?

→ MD5 cassable rapidement. Actions : forcer réinitialisation de tous les mdp. Futur : bcrypt/argon2 + salt, MFA.

Q2. Compte qui se verrouille sans fausse manoeuvre ?

→ Analyser logs (Event ID 4625 Windows). Peut être une attaque ou sync credentials (app mobile, ancien mdp).

Q3. Tester la robustesse des mdp AD légalement ?

→ Autorisation écrite de la direction, audit hors production, rapport confidentiel. Outils : John the Ripper, Hashcat.

Q4. Politique GPO de mots de passe pour une PME ?

→ 12 car min, complexité, historique 10 derniers, expiration 90j (ou sans si MFA), verrouillage après 5 tentatives, 15 min.

Q5. Credential stuffing vs force brute classique ?

→ Credential stuffing = couples login/mdp issus de fuites. Protection : MFA, détection IP anormales, Have I Been Pwned.

Types de cyberattaques

Malwares

TypeDescription et signes

Ver (Worm)Se propage automatiquement sans fichier hôte. Signes : saturation réseau.

Attaques sociales (ingénierie sociale)

  • Phishing : email frauduleux pour voler des identifiants. Reconnaître : URL suspecte, urgence, fautes.
  • Spear Phishing : phishing ciblé sur une personne avec infos personnalisées
  • Vishing / Smishing : phishing par téléphone ou SMS
  • Whaling : phishing ciblant des dirigeants (PDG, DAF)

Attaques réseau

  • DoS / DDoS : saturation d'un service (1 source vs botnet)
  • ARP / DNS Spoofing : base des attaques MitM
  • Injection SQL : insertion de code SQL dans un formulaire
  • XSS : injection de scripts malveillants dans une page web

Signes d'une cyberattaque : CPU/RAM élevés sans raison | processus inconnus | trafic inhabituel | connexions vers IP inconnues | mots de passe ne fonctionnant plus | tentatives d'authentification répétées dans les logs

Questions types d'examen

Q1. Email du 'directeur financier' demandant un virement de 25 000€ ?

→ BEC / Whaling. Signaux : urgence, demande inhabituelle. Mesures : double validation virements, DMARC, formation.

Q2. Saturation à 100% de la liaison Internet depuis 30 min ?

→ DDoS probable. Contacter FAI (scrubbing center), activer CDN/Cloudflare, analyser logs, activer PCA.

Q3. Fichiers renommés en '.locked' + README_DECRYPT.txt ?

→ Ransomware. Isoler les machines, identifier patient zéro, ne PAS payer, restaurer depuis sauvegardes saines.

Q4. Injection SQL, exemple concret et sécurisation ?

→ Saisie SQL dans un champ (' OR '1'='1). Requêtes préparées, validation entrées, WAF, moindre privilège DB.

Q5. Plan de sensibilisation phishing pour une PME ?

→ Formation + simulations de phishing + procédure de signalement + affichage + e-learning. Indicateur : taux de clics.

Chiffrement (Cryptographie)

Le chiffrement est la transformation de données lisibles en données illisibles à l'aide d'un algorithme et d'une clé. Seule la clé de déchiffrement permet de lire les données.

Chiffrement symétrique (1 clé)

AlgorithmeDescription
AES-256Standard actuel, très sécurisé. Utilisé dans HTTPS, WPA2, chiffrement de disques.
DES / 3DESAncien standard. DES est cassé (56 bits). 3DES obsolète.
ChaCha20Algorithme moderne, très rapide, utilisé dans TLS 1.3.

Chiffrement asymétrique (clé pub + privée)

  • Usage 1 — Confidentialité : je chiffre avec la clé publique du destinataire, seul lui déchiffre
  • Usage 2 — Signature : je signe avec ma clé privée, tout le monde vérifie avec ma clé publique
AlgorithmeDescription
RSAStandard historique. Clés 2048 ou 4096 bits. TLS, SSH, certificats.
ECDSA / ECDHCourbes elliptiques. Plus efficace que RSA à sécurité équivalente.
Diffie-HellmanÉchange de clé sécurisé sur un canal non sécurisé. Base de TLS.

Fonctions de hachage

  • SHA-256 / SHA-3 : standards actuels sécurisés (certificats, signatures, intégrité)
  • MD5 / SHA-1 : obsolètes et vulnérables — ne plus utiliser
  • Bcrypt / Argon2 : conçues pour hacher les mots de passe (lentes par conception)

Questions types d'examen

Q1. Mots de passe stockés en MD5, pourquoi problématique ?

→ MD5 cassable rapidement (rainbow tables, GPU). Solution : bcrypt ou argon2id + salt unique par utilisateur.

Q2. Comment fonctionne Diffie-Hellman ? Pourquoi dans TLS ?

→ DH établit un secret partagé sur un canal public. Utilisé pour générer la clé de session (Perfect Forward Secrecy).

Q3. Vérifier qu'un fichier n'a pas été altéré pendant le transfert ?

→ L'expéditeur calcule le hash SHA-256. Le destinataire recalcule et compare. Si identiques = intégrité OK.

Q4. Chiffrement d'un disque dur d'ordinateur portable d'entreprise ?

→ Protection en cas de vol/perte. BitLocker (Windows) avec AES-256 XTS. Clé de récupération dans AD. TPM.

Q5. Différence chiffrement vs encodage ?

→ Chiffrement : nécessite clé → sécurité. Encodage (Base64) : réversible sans clé → pas de sécurité.

12
Disponibilité

La disponibilité est l'un des 3 piliers de la sécurité (CIA). Elle garantit que les systèmes et données sont accessibles quand les utilisateurs en ont besoin.

Mesure de la disponibilité

DisponibilitéTemps d'arrêt annuel max
99% (2 neuf)~3,65 jours/an
99,9% (3 neuf)~8,7 heures/an
99,99% (4 neuf)~52 minutes/an
99,999% (5 neuf)~5 minutes/an (objectif datacenter tier 4)

Mécanismes

  • Haute disponibilité (HA) : redondance des composants pour éliminer les SPOF
  • Clustering : plusieurs serveurs en groupe, si un noeud tombe, les autres continuent
  • Load Balancing : répartition du trafic entre plusieurs serveurs
  • Failover automatique : basculement vers un système de secours
  • UPS : alimentation de secours en cas de coupure électrique
  • Redondance réseau : double FAI, liens redondants

Questions types d'examen

Q1. ERP disponible 99,9%. Temps d'arrêt max et architecture ?

→ 99,9% = 8,76h/an max. Architecture : 2 serveurs cluster actif/passif, VIP, stockage partagé, load balancer, UPS.

Q2. Identifier les SPOF : 1 routeur, 1 firewall, 1 switch, 2 serveurs ?

→ SPOF : routeur, firewall, switch (chacun unique). Éliminer : HSRP/VRRP, firewall HA, switch en stack. Les 2 serveurs sont OK.

Q3. Cluster actif/actif vs actif/passif ?

→ Actif/actif : tous traitent des requêtes (serveurs web). Actif/passif : secours en attente (firewall, serveur fichiers).

Q4. Alarme UPS, courant coupé. Actions ?

→ Vérifier autonomie, contacter électricien, prévenir utilisateurs. Si insuffisant : arrêt contrôlé dans l'ordre de priorité.

Q5. Justifier l'investissement HA auprès de la direction ?

→ Coût d'une heure d'arrêt (CA), pénalités SLA, perte de données, réputation. Comparer au coût de la solution HA.

13
Architecture multisites

Une architecture multisite répartit l'infrastructure informatique sur plusieurs sites géographiques distincts pour la disponibilité, la performance et la continuité d'activité.

Types de sites

TypeDescription
Actif/actifLes deux sites traitent des requêtes simultanément. Performances max. Complexité de synchronisation élevée.
Cold standbySite secondaire en attente. Temps de bascule long. Moins coûteux.
Warm standbyPartiellement en fonction, prêt à basculer rapidement.
Hot standbyEn production permanente, synchronisé en temps réel. Bascule quasi-instantanée.

Technologies d'interconnexion

  • VPN IPsec : tunnel chiffré via Internet. Économique, adapté aux petites structures.
  • MPLS : réseau privé opérateur, garanties QoS. Coûteux mais fiable.
  • SD-WAN : gestion intelligente des liens WAN multiples (Internet + MPLS). Optimisation automatique.
  • Liaisons louées (LS) : liens dédiés point à point. Haute dispo, faible latence, très coûteux.

Synchronisation des données

  • AD Sites et Services : réplication entre contrôleurs de domaine
  • DFS-R : réplication de partages de fichiers entre sites
  • SQL Always On, Oracle Data Guard, MySQL Replication
  • VMware vSphere Replication, Hyper-V Replica

Avantages

  • Continuité d'activité géographique
  • Proximité des utilisateurs (réduction latence)
  • Répartition de charge entre les sites
  • AD multisite : gestion centralisée + auth locale
  • Conformité RGPD (données dans la bonne région)

Inconvénients

  • Complexité de gestion
  • Coût des liaisons (MPLS)
  • Synchronisation à surveiller (conflits possibles)
  • Latence inter-sites impacte les applications sensibles
  • Chaque site = vecteur d'attaque à sécuriser

Questions types d'examen

Q1. Interconnecter siège + 5 agences de manière sécurisée et redondante ?

→ VPN IPsec avec 2 FAI (redondance), MPLS pour QoS, ou SD-WAN pour gérer les deux intelligemment.

Q2. AD multisite (Paris et Lyon), optimiser l'authentification ?

→ Sites et services AD : objets Site, sous-réseaux IP, liens de site (coût et schedule). DC locaux sur chaque site.

Q3. Lyon = plan de reprise pour Paris. Cold/warm/hot standby, quel RTO ?

→ Cold : RTO >24h. Warm : RTO 2-8h. Hot : RTO <1h (synchronisation temps réel). Choisir selon criticité et budget.

Q4. Réplication des partages de fichiers entre sites ?

→ DFS-R : namespace DFS, groupe de réplication, planification. Limites : conflits si fichier modifié sur 2 sites.

Q5. Comparer VPN IPsec, MPLS et SD-WAN pour 8 sites ?

→ VPN : économique, peu de QoS. MPLS : QoS garantie, coûteux. SD-WAN : combine les deux, routage intelligent. Recommander SD-WAN si budget moyen.
